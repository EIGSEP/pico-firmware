# PicoManager Operations Guide

PicoManager is a standalone service that owns all pico serial connections.
It discovers devices, monitors health, relays commands via Redis, and
uploads sensor data. Other processes (PandaClient, interactive scripts)
interact with picos exclusively through Redis.

## Running PicoManager

### Prerequisites

- Redis server running
- `eigsep_observing` installed (provides `EigsepRedis`)
- `picohost` installed (`pip install -e ./picohost`)
- `pico_config.json` present (generated by `flash-picos`)

### Standalone (development)

```bash
# From the pico-firmware directory
python -m picohost.manager --config pico_config.json --log-level DEBUG

# Or using the console script
pico-manager --config pico_config.json --log-level DEBUG
```

Press Ctrl+C to stop.

### systemd Service (production)

Install the service:

```bash
sudo cp picohost/pico-manager.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable pico-manager
sudo systemctl start pico-manager
```

Manage the service:

```bash
sudo systemctl status pico-manager   # check status
sudo systemctl restart pico-manager  # restart
sudo systemctl stop pico-manager     # stop
journalctl -u pico-manager -f        # tail logs
journalctl -u pico-manager --since "1 hour ago"  # recent logs
```

## Flashing Picos While Manager Is Running

The manager holds serial ports open, so you must stop it before flashing:

```bash
sudo systemctl stop pico-manager
flash-picos --uf2 build/pico_multi.uf2
sudo systemctl start pico-manager
```

## Sending Commands

### From Python

```python
from eigsep_observing import EigsepRedis
import json

r = EigsepRedis()

# Send a command to the RF switch
r.r.xadd("stream:pico_cmd", {
    "target": "switch",
    "cmd": json.dumps({"action": "switch", "state": "RFANT"}),
    "source": "interactive:myname",
})

# Send a motor command
r.r.xadd("stream:pico_cmd", {
    "target": "motor",
    "cmd": json.dumps({
        "action": "motor_command",
        "az_set_target_pos": 1000,
    }),
    "source": "interactive:myname",
})

# Set peltier temperature
r.r.xadd("stream:pico_cmd", {
    "target": "peltier",
    "cmd": json.dumps({
        "action": "set_temperature",
        "T_A": 25.0,
        "A_hyst": 0.5,
    }),
    "source": "interactive:myname",
})

# Send a raw command (no action dispatch)
r.r.xadd("stream:pico_cmd", {
    "target": "therm",
    "cmd": json.dumps({"some_firmware_key": 42}),
    "source": "interactive:myname",
})
```

### Reading Responses

```python
# Read responses (blocking, 5-second timeout)
result = r.r.xread({"stream:pico_resp": "$"}, block=5000, count=1)
```

## Device Claims

Claims provide visibility into which process is controlling a device.
They are advisory (soft) -- commands always execute regardless of claims.

```python
# Claim a device (TTL in seconds, auto-expires)
r.r.xadd("stream:pico_cmd", {
    "target": "switch",
    "cmd": json.dumps({"action": "claim", "ttl": 300}),
    "source": "switch_loop",
})

# Release a claim
r.r.xadd("stream:pico_cmd", {
    "target": "switch",
    "cmd": json.dumps({"action": "release"}),
    "source": "switch_loop",
})
```

If a command arrives from a source that doesn't match the current claim
owner, the command still executes but the response includes a `warning`
field.

## Checking Health

```python
import json

# Which picos are online?
r.r.smembers("picos")
# {'switch', 'motor', 'peltier', ...}

# Health details for a specific pico
health = json.loads(r.r.hget("pico_health", "switch"))
# {"connected": true, "last_seen": 1741234567.89, "app_id": 5}

# All health info
for name, data in r.r.hgetall("pico_health").items():
    print(name, json.loads(data))
```

## Redis Keys

| Key | Type | Description |
|-----|------|-------------|
| `picos` | set | Names of currently connected picos |
| `pico_health` | hash | Per-pico: `{connected, last_seen, app_id}` |
| `pico_config` | hash | Per-pico: `{port, app_id, usb_serial}` |
| `stream:pico_cmd` | stream | Commands to picos |
| `stream:pico_resp` | stream | Responses from picos |
| `pico_claim:{name}` | string | Current claim owner (TTL-based) |
| `metadata` | hash | Sensor data (unchanged) |

## Architecture

```
Clients (PandaClient, scripts)
    │
    │  Redis streams
    ▼
PicoManager (systemd service)
    │
    │  USB serial
    ▼
Pico devices (IMU, switch, peltier, motor, ...)
```

Only PicoManager holds serial connections. Multiple clients can
concurrently read pico data and send commands without serial port
contention.

## Troubleshooting

**Manager won't start**: Check `journalctl -u pico-manager` for errors.
Common issues: Redis not running, pico_config.json missing, serial ports
in use by another process.

**Pico shows as unhealthy**: Check physical USB connection. The manager
attempts automatic reconnection every 5 seconds.

**Commands not executing**: Verify the target device is in the `picos`
set. Check `stream:pico_resp` for error responses.

**Stale data**: If `last_seen` in `pico_health` is old, the pico may not
be sending status updates. Check physical connection and firmware.
